# RAG Culinary Assistant

Проект по Retrieval-Augmented Generation (RAG), в котором реализован кулинарный ассистент на русском языке.  
Он умеет искать релевантные рецепты в локальной базе, при необходимости добирать информацию из интернета и генерировать ответ с учетом истории диалога.

Вся основная логика здесь написана вручную, без тяжелых RAG-фреймворков.  
То есть это не история в духе “подключил готовую библиотеку и сделал вид, что что-то понял”, а нормальная самостоятельная реализация ключевых компонентов.

---

## Что делает проект

Ассистент отвечает на кулинарные вопросы, используя подход RAG:

- хранит рецепты в собственной векторной базе,
- разбивает длинные тексты на чанки,
- ищет релевантные документы по эмбеддингам,
- собирает контекст для языковой модели,
- генерирует ответ только на основе найденной информации,
- при слабом локальном совпадении пытается найти полезный контент в интернете,
- умеет учитывать историю диалога через переформулирование текущего запроса.

Если по-человечески: пользователь спрашивает что-то про блюдо, система ищет подходящие рецепты, собирает из них контекст и уже потом просит модель ответить, а не выдумывать ерунду с потолка.

---

## Структура решения

В проекте реализованы следующие компоненты:

### 1. Локальная база рецептов
В качестве источника данных используется датасет `povarenok.csv` с рецептами.  
Для ускорения в ноутбуке берутся первые 10 000 записей.

Каждая запись содержит:
- название рецепта,
- ингредиенты,
- текст рецепта.

### 2. Разбиение текста на чанки
Так как рецепты бывают длинными, каждый текст разбивается на небольшие куски.  
Используется рекурсивное разбиение по уровням:

- `\n\n`
- `\n`
- `" "`
- `""`

Для некоторых уровней добавляется overlap, чтобы не терять смысл на границах чанков.

### 3. Векторная база с LSH
Для быстрого поиска похожих текстов реализован собственный `LSHIndex`.

Он:
- принимает эмбеддинги чанков,
- строит несколько хеш-таблиц,
- по запросу возвращает ближайшие кандидаты.

Это не идеальный production-grade ANN-движок, но для учебного проекта вполне нормальный и понятный вариант, особенно когда хочется показать, что ты понимаешь механику поиска, а не просто дергаешь готовый FAISS.

### 4. Эмбеддинги
Для кодирования текстов используется модель:

- `intfloat/multilingual-e5-large`

Она хорошо подходит для семантического поиска и умеет работать с русским языком.

### 5. Генерация ответа
Для генерации ответа используется:

- `Qwen/Qwen2-1.5B-Instruct`

Модель получает:
- системный промпт,
- пользовательский вопрос,
- найденный контекст из рецептов.

После этого генерируется итоговый ответ на русском языке.

### 6. Поиск в интернете
Если локальная база не находит достаточно релевантных документов, система пробует добрать информацию из веба:

- делает поисковый запрос,
- проходит по найденным ссылкам,
- вытаскивает текст страниц,
- сравнивает их релевантность с запросом,
- подставляет лучшие куски в контекст.

Идея простая: если в датасете нет нужного блюда или факта, можно не делать вид, что всё знаешь, а попробовать реально что-то найти.

### 7. История диалога
Чтобы можно было задавать уточняющие вопросы вроде:

- “А сколько в нем калорий?”
- “А можно без мяса?”
- “А при какой температуре?”

система хранит историю общения и переформулирует новый вопрос в самодостаточную форму.

Например:
- было: `Какая у него калорийность?`
- стало: `Какая калорийность у салата Цезарь?`

Это позволяет учитывать историю без тупого запихивания всего чата в промпт.

## Как запустить

1. Положить файл `povarenok.csv` рядом с ноутбуком.
2. Открыть `RAG.ipynb`.
3. Последовательно выполнить ячейки:

   * загрузка данных,
   * загрузка моделей,
   * построение эмбеддингов,
   * индексация чанков,
   * генерация ответов,
   * сценарии тестирования.

## Примеры запросов

Ассистент рассчитан на вопросы такого типа:

* Как приготовить классический борщ?
* Сделай простой рецепт ужина с курицей и картофелем.
* Можно сделать борщ без мяса?
* Сколько примерно калорий в порции блюда?
* Как приготовить простой безалкогольный коктейль?

Также проверяются более сложные сценарии:

* уточняющие вопросы по истории,
* редкие или странные запросы,
* запросы, которых нет в локальной базе.

Что здесь реально получилось хорошо:

* понятная и прозрачная архитектура RAG;
* собственная реализация ключевых компонентов;

Тут есть и retrieval, и генерация, и попытка честно решить ограничения обычного RAG.

---

## Ограничения

Без них, конечно, никуда.

### 1. Качество зависит от датасета

Если нужного рецепта или нужного факта нет в `povarenok.csv`, локальный поиск не спасет.

### 2. Модель всё еще может галлюцинировать

Даже с RAG генеративная модель иногда отвечает не по сути или добавляет лишнее.

### 3. Веб-поиск не гарантирует качество

Интернет-источники могут быть шумными, мусорными или нерелевантными.

### 4. История диалога помогает, но не магия

Переформулирование запроса улучшает понимание контекста, но не исправляет все ошибки модели.

### 5. Это ноутбук, а не production-сервис

Проект хорошо подходит для демонстрации идеи, экспериментов и учебной работы, но до полноценного сервиса его еще надо нормально допиливать.

## Что можно улучшить

Если развивать проект дальше, то логичные направления такие:

* заменить LSH на более сильный индекс поиска;

* улучшить веб-скрейпинг и фильтрацию источников;
* нормализовать и чистить тексты рецептов;
* сделать отдельный API/интерфейс;
* добавить автоматическую оценку качества retrieval и generation;
* протестировать более сильную генеративную модель.
